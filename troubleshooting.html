Troubleshooting

This page covers the four most common problems people run into with PVM, along with diagnostic commands and where to get help.

Project Detection

Problem: PVM does not recognize a directory as a project and falls back to global settings.

Why it happens: PVM identifies a project by looking for one of several marker files in the current directory or its parents. If none are present, there is no project context to detect.

Solution: Add at least one of the following to your project root:

  • .perl-version — a plain-text file containing the Perl version string (e.g. 5.40.2)
  • cpanfile — a module dependency file
  • pvm.toml — a full PVM project configuration file
# Quickest fix: pin the Perl version
echo "5.40.2" > .perl-version

To confirm PVM now sees the project:

pvm current --detailed

Module Installation

Problem: Module installation fails, or installed modules are not found at runtime.

Why it happens: Modules are installed relative to a project context. If PVM cannot detect a project (see above), it may install to the wrong location, or the runtime may pick up a different Perl than the one modules were installed for.

Solution: First confirm you are in a recognized project directory, then run the workspace doctor:

pvm workspace status --json
pvm self doctor

The workspace status output shows the active Perl version, module search paths, and any configuration inconsistencies. The self doctor command checks PVM's own installation and shell integration.

Once the project context is correct, reinstall the missing module:

pm install Module::Name

Build Issues

Problem: Build fails with type errors or unexpected PSC diagnostics.

Why it happens: PSC infers types from your Perl source without requiring annotations. If the configuration is not pointing at the right files, or if the inferred types do not match what you expect, the checker will report errors.

Solution: Check your PSC configuration and run the checker directly against the file in question:

psc check lib/

PSC will report the inferred type alongside any mismatch. For a given file, psc analyze provides a more detailed breakdown:

psc analyze lib/MyModule.pm

If the errors appear in files you did not change, run the workspace doctor to rule out environment drift:

pvm self doctor

See the Type Checking guide for a full explanation of how PSC infers types and how to interpret its diagnostics.

Environment Issues

Problem: The wrong Perl version is active — scripts run under a different version than expected.

Why it happens: PVM resolves the active Perl version by walking up the directory tree looking for .perl-version or pvm.toml, then falling back to the global default. If shell integration is not set up correctly, the PATH may bypass PVM entirely.

Solution: Start by checking what version PVM thinks is active and how it arrived at that choice:

pvm current --detailed
pvm versions

pvm current --detailed shows the resolved version and the source file that set it. pvm versions lists every installed version so you can confirm the one you want is present.

If PVM is being bypassed entirely, the shell integration is likely missing. Add the following to your shell startup file (~/.bashrc, ~/.zshrc, or equivalent) and open a new shell:

eval "$(pvm init)"

After reopening the shell, verify the path is correct:

which perl
perl -v

Diagnostic Commands

These commands are useful starting points for any PVM problem:

pvm self doctor
Checks PVM's installation, shell integration, and configuration for common problems.
pvm self doctor --fix
Attempts to automatically correct the issues self doctor finds.
pvm workspace status
Shows the active project context, Perl version, and module paths.
pvm workspace status --json
Same information in machine-readable form, useful for scripts and bug reports.
pvm current --detailed
Shows the resolved Perl version and the file or setting that determined it.
pvm versions
Lists all installed Perl versions.
pvm --verbose [command]
Runs any PVM command with verbose output. Add this flag to any failing command to see more detail.

Getting Help

If the steps above do not resolve your issue, the most useful thing to bring to a bug report is the output of:

pvm workspace status
pvm self doctor

Report issues at https://github.com/perigrin/pvm/issues. Include the verbose output of the failing command and the output of the two diagnostic commands above.
